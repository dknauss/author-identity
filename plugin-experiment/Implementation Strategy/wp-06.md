WORK PACKAGE 06: Content rights and AI consent

Goal:
- Give publishers and individual authors machine-readable control over AI training consent and content rights, expressed in HTML meta tags, HTTP headers, feed metadata, and `ai.txt`.

Primary targets:
- `inc/rights.php`
- `tests/phpunit/test-rights.php`

Depends on:
- WP-01 (adapter interface for per-author consent resolution).

Problem statement:
- Publishers and writers increasingly need to express preferences about AI training use of their content. Current mechanisms (`robots.txt`, `ai.txt`, TDM headers) are site-wide. No WordPress plugin offers per-author or per-post AI consent that flows into structured metadata across feeds and HTML. On multi-author sites, different authors may have different preferences — a staff reporter may want maximum reach while a guest contributor opts out.

Planned changes:

1. Register per-author AI consent user meta:
   ```php
   register_meta( 'user', 'byline_feed_ai_consent', [
       'type'         => 'string',
       'single'       => true,
       'show_in_rest' => true,
       'default'      => '',
   ] );
   ```
   Values: `'allow'`, `'deny'`, `''` (unset, inherits site default).

2. Register per-post AI consent override:
   ```php
   register_post_meta( '', '_byline_ai_consent', [
       'type'         => 'string',
       'single'       => true,
       'show_in_rest' => true,
       'default'      => '',
   ] );
   ```
   Values: `'allow'`, `'deny'`, `''` (unset, inherits from author).

3. Consent resolution logic:
   ```php
   function byline_feed_resolve_ai_consent( \WP_Post $post ) : string {
       // Post-level override takes precedence
       $post_consent = get_post_meta( $post->ID, '_byline_ai_consent', true );
       if ( in_array( $post_consent, [ 'allow', 'deny' ], true ) ) {
           $consent = $post_consent;
       } else {
           // Fall back to most restrictive author preference
           $authors = byline_feed_get_authors( $post );
           $consents = array_filter( array_column( $authors, 'ai_consent' ) );
           if ( in_array( 'deny', $consents, true ) ) {
               $consent = 'deny';
           } elseif ( in_array( 'allow', $consents, true ) ) {
               $consent = 'allow';
           } else {
               $consent = '';
           }
       }
       return apply_filters( 'byline_feed_ai_consent', $consent, $post );
   }
   ```
   When multiple authors have conflicting preferences, the most restrictive preference wins (deny beats allow). This is the safe default — an author's opt-out should not be overridden by a co-author's opt-in.

4. HTML meta tag output (`wp_head` hook):
   - If resolved consent is `deny`:
     ```html
     <meta name="robots" content="noai, noimageai" />
     ```
   - Note: `noai` and `noimageai` are Google's conventions. Other crawlers may use different directives. Output additional directives as conventions emerge.

5. HTTP header output:
   - Hook `send_headers` or `wp_headers` for singular posts.
   - If resolved consent is `deny`, add TDM-Reservation header:
     ```
     TDMRep: https://example.com/tdm-policy
     ```
   - The policy URL can be configured in plugin settings or default to a generated page.

6. Feed-level rights metadata:
   - In RSS2 item output (extending WP-02), if consent is `deny`:
     ```xml
     <cc:license>https://example.com/tdm-policy</cc:license>
     ```
   - Or a Byline extension element if the spec evolves to include rights.
   - If consent is `allow` and a Creative Commons license is configured, output `<cc:license>` with the license URL.

7. `ai.txt` generation:
   - Register a rewrite rule for `/ai.txt`.
   - Generate the file dynamically from plugin settings:
     ```
     User-Agent: *
     Disallow: /
     ```
   - Or allow granular configuration per content type.
   - Settings page with a simple textarea override for full manual control.

8. User profile field:
   - Add "AI Training Consent" field to user profile edit screen.
   - Radio buttons: "Allow", "Deny", "No preference (inherit site default)".
   - Description: "Controls whether your content may be used for AI model training. When set to Deny, machine-readable signals are added to pages and feeds containing your work."

9. Post editor control:
   - Add to the block editor sidebar (could share the `PluginDocumentSettingPanel` with the perspective control from WP-03).
   - Select: "Inherit from authors", "Allow AI training", "Deny AI training".

10. Write PHPUnit tests (`tests/phpunit/test-rights.php`):
    - Single author, consent=deny: `noai` meta tag present, TDM header present.
    - Single author, consent=allow: no `noai` meta tag, no TDM header.
    - Single author, consent unset: no output (inherits nothing).
    - Multiple authors, one deny + one allow: resolved consent is deny.
    - Post-level override beats author-level preference.
    - Filter overrides everything.
    - `ai.txt` endpoint returns expected content.
    - Feed items for denied posts include rights metadata.

Validation:
- User profile shows AI consent field.
- Post editor shows AI consent override.
- Singular pages for denied posts include `<meta name="robots" content="noai, noimageai">`.
- HTTP response headers include TDM-Reservation for denied posts.
- Feed output includes rights metadata for denied posts.
- `/ai.txt` returns valid content.
- Multi-author conflict resolution correctly applies most-restrictive-wins.
- PHPUnit suite green.
- PHPCS clean.

Notes:
- `noai` and `noimageai` are not universally standardized. They are advisory signals that crawlers may or may not honor. The plugin documentation must be clear about this: these are consent signals, not enforcement mechanisms.
- The TDMRep specification is emerging (W3C). Implementation should track the spec and update header format as it stabilizes.
- Per-author consent in structured metadata is genuinely novel. Nobody is doing this. It will attract attention from the journalism and rights management communities. The documentation and communication around this feature should be precise about what it does and does not guarantee.
- This work package can be developed in parallel with WP-02 through WP-05 after WP-01 is complete, but the feed integration (step 6) depends on WP-02 being in place.

Retroactive consent and identity severability:

The AI consent mechanism must support retroactive changes. An author who initially published with consent=allow (or unset) must be able to change their preference to deny, and that change must propagate:

1. Changing the user meta `byline_feed_ai_consent` from `allow` to `deny` should immediately affect all future page loads and feed generations for every post attributed to that author. No per-post migration is needed — the resolution logic already queries author meta at render time.

2. The `ai.txt` generation should reflect the most current consent state. If an author withdraws consent, their content should appear in the deny list on the next `ai.txt` generation.

3. Feed output should reflect updated consent on the next feed build. Any system re-crawling the feed will pick up the changed `noai` signal. This won't reach systems that already ingested the content at crawl time, but it establishes a dated, machine-readable record of when consent was withdrawn — which has evidentiary value.

4. If an author is removed from a post's attribution entirely (identity severability — see "The permanence problem" in `author-identity-vision.md`), the consent resolution should fall back to the remaining authors' preferences or the site default. The removed author's `deny` should no longer apply to a post they are no longer attributed to, because the consent signal is tied to the authorship relationship, not the content itself.

5. The plugin should log consent state changes (author, timestamp, old value, new value) in a way that provides an audit trail. If a legal or compliance question arises about when consent was withdrawn, the log answers it. This log should be accessible to site administrators but not exposed in public metadata.

Design principle: every piece of identity and consent metadata the plugin emits should be reversible from the origin server. See "The permanence problem" section in the vision doc for the full rationale on why this matters for author safety, privacy, and the right to be forgotten in federated/syndicated contexts.
