WORK PACKAGE 02: Byline RSS2, Atom, and JSON Feed output

Goal:
- Output valid Byline spec elements in WordPress RSS2, Atom, and JSON Feed output using the normalized author data from WP-01's adapter layer.

Primary targets:
- `inc/feed-rss2.php`
- `inc/feed-atom.php`
- `inc/feed-json.php`
- `tests/phpunit/test-feed-rss2.php`
- `tests/phpunit/test-feed-atom.php`
- `tests/phpunit/test-feed-json.php`

Depends on:
- WP-01 (adapter interface and public API).

Problem statement:
- WordPress feeds contain minimal author information: RSS2 has `<author>` (email only by spec) and `<dc:creator>` (plain text). Atom has `<author>` with name/URI/email. None provide structured identity, perspective, role, or profile links. Multi-author posts are reduced to the primary author or a comma-separated name list.

Planned changes:

1. RSS2 output (`inc/feed-rss2.php`):

   Hook `rss2_ns` to declare the Byline namespace:
   ```php
   add_action( 'rss2_ns', function() {
       echo 'xmlns:byline="https://bylinespec.org/1.0"' . "\n";
   } );
   ```

   Hook `rss2_head` to output `<byline:contributors>` containing a `<byline:person>` for each unique author across all posts in the current feed query:
   - Collect all authors from all posts in the current feed loop (use `$wp_query->posts`).
   - Deduplicate by `id`.
   - For each author, output:
     ```xml
     <byline:person id="{id}">
       <byline:name>{display_name}</byline:name>
       <byline:context>{description, capped at 280 chars}</byline:context>
       <byline:url>{url}</byline:url>
       <byline:avatar>{avatar_url}</byline:avatar>
       {for each profile: <byline:profile href="{href}" rel="{rel}"/>}
       {if now_url: <byline:now>{now_url}</byline:now>}
       {if uses_url: <byline:uses>{uses_url}</byline:uses>}
     </byline:person>
     ```
   - Omit empty optional elements (don't output `<byline:context/>` for empty bios).
   - Fire `byline_feed_after_rss2_contributors` action.

   Hook `rss2_item` to output per-item elements:
   - For each author attributed to the current post:
     ```xml
     <byline:author ref="{id}"/>
     <byline:role>{role}</byline:role>
     ```
   - If perspective is set for the post (from meta or filter):
     ```xml
     <byline:perspective>{perspective}</byline:perspective>
     ```
   - Fire `byline_feed_after_rss2_item` action.
   - Apply `byline_feed_item_xml` filter to the complete per-item output.

2. Atom output (`inc/feed-atom.php`):
   - Use `atom_ns` hook to declare Byline namespace on the `<feed>` element.
   - Use `atom_head` hook to output `<byline:contributors>` block at feed level (same as RSS2).
   - Use `atom_entry` hook for per-entry Byline namespace extensions: `<byline:author ref="..."/>`, `<byline:role>`, and `<byline:perspective>`, matching the RSS2 approach.
   - Apply `byline_feed_atom_entry_xml` filter to the complete per-entry Byline output.

   **Current shipped behavior:** The plugin adds Byline namespace elements alongside WordPress's default Atom `<author>` output. Standard Atom elements are left in place. Byline output is purely additive.

   **Future enhancement (not yet implemented):** Atom natively supports multiple `atom:author` and `atom:contributor` elements per entry (RFC 4287 Person Constructs with name, uri, email). A future pass could replace WordPress's single default `<author>` with multi-author `<author>` elements and use `<contributor>` for editors/researchers, giving Atom-aware parsers structured multi-author data even without Byline namespace support. This would make Atom the richest multi-author output format. Example of the aspirational two-author entry:
     ```xml
     <entry>
       <title>Example Post</title>
       <author>
         <name>Jane Doe</name>
         <uri>https://example.com/author/jdoe</uri>
       </author>
       <author>
         <name>Alex Smith</name>
         <uri>https://example.com/author/asmith</uri>
       </author>
       <contributor>
         <name>Pat Editor</name>
         <uri>https://example.com/author/peditor</uri>
       </contributor>
       <byline:author ref="jdoe"/>
       <byline:author ref="asmith"/>
       <byline:role>staff</byline:role>
       <byline:perspective>reporting</byline:perspective>
     </entry>
     ```

3. Preserve standard elements:
   - In RSS2: do NOT remove or modify existing `<author>`, `<dc:creator>` elements. Byline output is purely additive.
   - In Atom: Byline namespace elements are additive alongside WordPress's default Atom `<author>` output. Standard Atom elements are not modified or replaced.
   - Byline namespace elements are always additive in both formats.

4. XML escaping:
   - All text content MUST be escaped via `esc_xml()` (WP 5.5+) or `esc_html()` as fallback.
   - URLs MUST be escaped via `esc_url()`.
   - Author descriptions containing HTML MUST be stripped to plain text before output.

5. Write PHPUnit tests:
   - `test-feed-rss2.php`:
     - Fetch RSS2 feed output via `do_feed_rss2` or similar.
     - Parse XML, verify namespace declaration.
     - Verify `<byline:contributors>` block contains correct `<byline:person>` elements.
     - Verify per-item `<byline:author>` refs match contributors.
     - Verify standard `<author>` elements are still present.
     - Test with single author, multiple authors, guest authors.
     - Test with and without perspective set.
     - Test that empty optional fields are omitted, not output as empty elements.
   - `test-feed-atom.php`: parallel tests for Atom output.

Validation:
- RSS2 feed output is well-formed XML.
- Byline namespace is declared on `<rss>` element.
- `<byline:contributors>` appears in `<channel>` with correct person elements.
- Each `<item>` has `<byline:author ref="..."/>` matching a contributor.
- Standard feed elements are unmodified.
- Feed output passes the Byline validator at bylinespec.org/tools/validator (if available).
- PHPUnit suite green for all feed scenarios.
- No PHPCS violations.

Notes:
- The `rss2_head` hook fires inside `<channel>` after standard channel elements. The `rss2_item` hook fires inside each `<item>`. These are standard WordPress hooks.
- The contributors block requires iterating all posts in the feed to collect unique authors. This is a pre-pass before the feed loop. Cache the result for the duration of the feed request.

### JSON Feed output (`inc/feed-json.php`)

JSON Feed is the most natural format for Byline's data shape because JSON Feed 1.1 already has an `authors` array at both feed-level and item-level, and JSON Feed's extension mechanism (underscore-prefixed properties) is simpler than XML namespacing.

The module operates in two modes:

1. **Plugin integration mode:** If a JSON Feed plugin is active (detected by `json_feed_item` filter or `json_feed_init` function), hooks into its pipeline via `json_feed_item` and `json_feed_channel` filters to enrich existing output.
2. **Standalone mode:** If no JSON Feed plugin is found, registers a `/feed/json` endpoint via `add_feed( 'json', ... )` and renders a complete JSON Feed 1.1 document.

Both modes use the same builder functions and produce identical `_byline` extension objects.

**Extension mapping:**

Standard JSON Feed `authors` fields (`name`, `url`, `avatar`) carry data that any JSON Feed reader already parses. The `_byline` extension object on each author entry carries Byline-specific enrichment:

```json
{
  "name": "Jane Doe",
  "url": "https://example.com/author/jdoe",
  "avatar": "https://example.com/avatars/jdoe.jpg",
  "_byline": {
    "id": "jdoe",
    "context": "Staff writer covering technology.",
    "role": "staff",
    "profiles": [
      { "href": "https://mastodon.social/@jdoe", "rel": "me" }
    ],
    "now_url": "https://jdoe.com/now",
    "uses_url": "https://jdoe.com/uses"
  }
}
```

Item-level perspective is a `_byline` object on the item itself:

```json
{
  "id": "https://example.com/post/123",
  "title": "Example Post",
  "authors": [...],
  "_byline": {
    "perspective": "reporting"
  }
}
```

Feed-level metadata includes spec version and organization:

```json
{
  "version": "https://jsonfeed.org/version/1.1",
  "_byline": {
    "spec_version": "1.0",
    "org": {
      "name": "The Daily Example",
      "url": "https://example.com"
    }
  }
}
```

**Tests (`test-feed-json.php`):**
- Standalone `/feed/json` endpoint returns valid JSON with `version` field.
- Feed-level `authors` array contains deduplicated author entries with `_byline` extensions.
- Per-item `authors` array contains correct author entries with roles in `_byline.role`.
- Item-level `_byline.perspective` present when set, absent when unset.
- Standard JSON Feed fields (`title`, `home_page_url`, `feed_url`, `items`) are present.
- `_byline.org` contains site name and URL.
- Empty optional fields are omitted from `_byline` objects.

The JSON Feed mapping is the plugin's proposed alternative structure. The Byline spec repo already contains a JSON Feed example (`corporate-blog.json`) with a different model — see the spec-alignment issues in [byline-spec-plan.md](../docs/planning/byline-spec-plan.md) for the current divergences.
